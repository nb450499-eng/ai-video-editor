## 0. 步骤 0 的产出与验收

**目标**：为第一阶段建立可驱动指标的真值集，并产出标注所需的目录规范、shot_spec 和轻量标注工具。

**最终交付物**：
1. 素材按 §1 规范归档（固定 asset_id）。
2. 每个镜头一份 `shot_spec.json`（§2）。
3. 每条 take 一份 `*.truth.json`（§3），含 0–N 个可用窗口 + 事件点 + 分级。
4. 自研轻量标注页（§4），导出即符合 §3 schema。
5. ≥ 20 条 take 真值入库，其中 **≥ 3 条「无可用窗口（应拒绝）」** 样本。
6. 关键样本双人标注，一致性达 §5 门槛。

**验收清单**：
- [ ] 目录/命名 100% 符合 §1，asset_id 唯一且稳定。
- [ ] 每镜头 shot_spec 通过 Pydantic 校验。
- [ ] ≥ 20 条 take truth，含 ≥ 3 条应拒绝样本。
- [ ] 双标注一致性：窗口 IoU 中位 ≥ 0.8，事件点差中位 ≤ 200ms。
- [ ] 数据集已划分 dev / val / locked-test，且 locked-test 单独封存。

## 1. 素材目录与命名规范（从零建立）

```
projects/
└── <project_id>/                # 例：tiktok_wirestripper_0624
    ├── manifest.json            # 项目级：产品、镜头清单、素材根
    ├── shots/
    │   ├── 01/
    │   │   ├── shot_spec.json
    │   │   ├── take_001.mp4      # 原始素材，勿改
    │   │   ├── take_002.mp4
    │   │   └── ...
    │   ├── 02/
    │   └── ...
    └── truth/                    # 真值与标注产物
        ├── 01/
        │   ├── take_001.truth.json
        │   └── ...
        └── split.json           # dev/val/locked-test 划分
```

**命名硬规则**：
- `shot_id`：两位零填充字符串 `"01"…"06"`。
- take 文件：`take_NNN.mp4`，三位零填充，序号在镜头内唯一。
- **asset_id = `<shot_id>/take_NNN`**（如 `03/take_002`），全局唯一、永不复用、删除不补号。
- 原始素材只读；任何分析/代理/截取都不写回原文件。

**manifest.json（最小）**：
```json
{
  "project_id": "tiktok_wirestripper_0624",
  "product": "self-strip wire tool",
  "shots_root": "shots",
  "truth_root": "truth",
  "shot_ids": ["01","02","03","04","05","06"]
}
```

## 2. shot_spec 编写规范（标注的判定基准）

**为什么先写 spec**：真值里的"必要事件覆盖""禁止画面""结果停留时长"都以 spec 为准；没有 spec，标注就没有客观标尺。每个镜头**先填 spec，再标 take**。

字段语义见开发方案 §3.1。编写要点：
- `required_events`：列出该镜头**有效窗口必须覆盖**的动作链事件，用稳定英文 key（如 `tool_grips_wire / pull_starts / insulation_removed / result_visible`）。同一动作链全项目用词统一。
- `required_states`：动作前/后必须满足的可见状态（如"动作前外皮完整""动作后线芯清晰"）。
- `forbidden`：硬淘汰画面（如"动作从中间开始""结果被手遮挡""不同电线伪装成连续动作"）。
- `timing.min_result_hold_ms`：结果至少持续多久才算成立（如 500）。**这是判定 D 级/拒绝的关键阈值。**
- `role` 决定事件模板：`product_demo` 需完整动作链；`product_closeup` 可无 ACTION_PEAK；`before_after` 必含 BEFORE_STATE / TRANSFORMATION / AFTER_STATE。

示例直接复用文档 §4 的 shot_03 JSON。

## 3. 真值标注规范（`*.truth.json`）

### 3.1 事件状态机（标注者心智模型）
```
PREPARE_INVALID → READY_VISIBLE → ACTION_START → ACTION_PEAK/CONTACT
→ RESULT_FIRST_VISIBLE → RESULT_HOLD → TAIL_INVALID
```

**本阶段每条可用窗口必须标的 3 个事件点**（毫秒）：
- `action_start_ms`：核心动作**真正开始**的首帧（不是手进画面，是动作启动）。
- `result_first_visible_ms`：结果**第一次清晰可见**的帧。
- `result_hold_end_ms`：结果**清晰且稳定**保持的结束帧（被遮挡/移出/切走即终止）。

### 3.2 窗口边界判定规则
- `source_in_ms`：在 `action_start_ms` 之前留自然前摇，落在 READY_VISIBLE 区间内、**剔除** PREPARE_INVALID。原则：开头帧不得是模糊/找位置/空镜。
- `source_out_ms`：原则上 = `result_hold_end_ms`（可含少量自然 post_roll），**剔除** TAIL_INVALID。结尾帧必须是清晰结果，不得是动作未完成或遮挡帧。
- 结果停留 = `result_hold_end_ms − result_first_visible_ms`，须 ≥ spec.`min_result_hold_ms`，否则该窗口判 **不可用**。
- 一条 take 可有 **多个**可用窗口（不同 take-after 或不同动作）；也可有 **0 个**。

### 3.3 分级（人工，与系统 A/B/C/D 对齐）
- A：动作/结果/构图/画质均好，边界清晰。
- B：可用，仅轻微缺陷或需轻微调速。
- C：明显风险，需人工判断，默认不自动进成片。
- D / 无窗口：关键事件缺失、严重模糊/遮挡、结果时长不足、不可剪 → 应拒绝。

### 3.4 truth.json schema
```json
{
  "asset_id": "03/take_002",
  "shot_id": "03",
  "duration_ms": 5180,
  "usable": true,
  "windows": [
    {
      "window_id": "w1",
      "source_in_ms": 1420,
      "source_out_ms": 3710,
      "events": {
        "action_start_ms": 2040,
        "result_first_visible_ms": 2910,
        "result_hold_end_ms": 3710
      },
      "grade": "A",
      "notes": "动作完整，结果清晰保持790ms，关键区间无遮挡"
    }
  ],
  "reject_reasons": [],
  "annotator": "annotatorA",
  "annotated_at": "2026-06-24T16:40:00+08:00"
}
```
**应拒绝样本**：`usable=false`、`windows=[]`、`reject_reasons` 用 §6.1 标准枚举（如 `"result_hold_too_short"`, `"key_occluded"`, `"event_missing:result_visible"`, `"severe_blur"`）。

> 时间一律整数毫秒。多窗口 `window_id` 在 take 内唯一。

## 4. 自研轻量标注页 · 实现规格（交给 codex）

**形态**：本地单页应用，零部署优先。推荐 **单 HTML + 原生 JS**（`<video>` 播放），或极简 Flask 仅做读目录/存盘。无需数据库。

**核心要求**：
- 帧级定位：基于已知帧率，`←/→` 单帧步进；`Shift+←/→` 跳 10 帧。时间显示到毫秒。
- 加载：选 `projects/<id>`，左侧列出 take 列表（已标/未标状态着色），右侧播放器。
- 同屏展示该镜头 `shot_spec` 的 `required_events / forbidden / min_result_hold_ms`，方便对照判定。
- 打点（热键）：
  - `I` = 设 `source_in`，`O` = 设 `source_out`
  - `A` = `action_start`，`R` = `result_first_visible`，`E` = `result_hold_end`
  - `N` = 新建一个窗口，`D` = 删除当前窗口
  - `X` = 标记本 take「无可用窗口（应拒绝）」并弹出 reject_reasons 多选
  - `G` = 设当前窗口分级 A/B/C/D
  - `S` = 保存为 `truth/<shot>/take_NNN.truth.json`
- 校验：保存前检查 `source_in < action_start < result_first_visible ≤ result_hold_end ≤ source_out`，结果停留 < spec 阈值时高亮提醒（不阻断，记 grade=D 由人判断）。
- 导出格式**严格等于 §3.4 schema**，可被 Pydantic 直接读。
- 双标注：同一 take 可由不同 annotator 各存一份（文件名加 annotator 后缀），供 §5 比对。

**交付**：`tools/annotator/`（页面 + README），一条命令本地起；附 3 条样例 truth.json 自测。

## 5. 标注质量控制

- 至少 **2 名标注者**；其中关键样本（每镜头抽 1–2 条 + 全部应拒绝样本）双标。
- 一致性门槛：可用窗口 **IoU 中位 ≥ 0.8**；三个事件点 **差值中位 ≤ 200ms**。
- 不达标 → 开一次对齐会统一判定口径，更新本手册「判定细则」附录，重标分歧样本。
- 有争议且对齐后仍分歧的样本：单独归类，**不作为锁定测试集的绝对真值**。

---

## 6. 数据划分与红线

- 在 `truth/split.json` 把镜头/take 划成 **dev / val / locked-test**（建议 6:2:2）。
- **locked-test 封存**：禁止用于调 prompt、调阈值、调权重；只在阶段验收时跑一次。
- 应拒绝样本在三个集合中都要有分布，避免拒绝能力只在某一集体现。

## 7. 给 codex 的开工顺序

1. 建 §1 目录骨架 + `manifest.json` 模板 + `manifest/schemas.py` 里的 `ShotSpec` / `TruthFile` Pydantic 模型（供标注页校验复用）。
2. 实现 §4 标注页（先跑通 I/O/A/R/E/N/S + 导出，再补校验与分级）。
3. 人工对首批镜头先写 `shot_spec.json`，再标注。
4. 标够 ≥ 20 条（含 ≥ 3 应拒绝）→ 跑 §5 一致性 → 写 `split.json` → 步骤 0 验收。

完成步骤 0 后进入开发方案步骤 1（媒体标准化）。
